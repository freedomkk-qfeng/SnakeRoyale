# SnakeRoyale 设计说明

当前版本：`0.3.0`

## 目标

SnakeRoyale 是一个面向 AI 编程教学的轻量级多人贪吃蛇对战平台。

设计目标主要有五个：

1. 部署足够简单，适合课堂快速拉起。
2. 协议足够直观，学生可以很快读懂并写出自己的客户端。
3. 服务端保持权威模拟，确保规则一致且可控。
4. 通过 Dashboard 和状态接口提供足够高的可观测性。
5. 网络层足够稳健，避免单个慢客户端拖累全场。

## 总体架构

项目运行时主要分成三部分：

1. `server/`
   负责游戏主循环、注册接口、WebSocket 游戏连接、观战连接和 Dashboard 静态资源。
2. `client/`
   提供示例 AI 客户端和批量启动脚本，便于课堂演示和冒烟测试。
3. `server/static/`
   提供 Dashboard，用于展示实时战局、排行榜和生存统计。

整个系统以服务端为唯一事实来源。

## 核心运行模型

### 服务端权威 Tick Loop

游戏按离散 tick 推进。每个 tick 服务端会依次完成：

1. 应用玩家待生效方向。
2. 计算下一步蛇头位置。
3. 处理撞墙、撞身体、头对头碰撞。
4. 更新蛇身、食物、分数和历史记录。
5. 生成并广播状态快照。

这个模型的重点不是追求极致平滑，而是让教学环境下的规则和行为简单、稳定、可解释。

### 全量状态快照协议

当前协议选择发送全量状态，而不是增量差分。

这样做的原因：

1. 更容易理解和调试。
2. 协议实现成本低，适合教学。
3. 断线恢复和 Dashboard 渲染更直接。

代价是：

1. 带宽开销更大。
2. 在课堂无线网络较弱时，需要谨慎调节 tick 频率。

## 网络设计

### 独立 Sender Task

当前网络层已经改成“每个连接独立 sender task + mailbox”。

主 game loop 不再直接按顺序给每个 WebSocket 发消息，而是只把最新消息批次塞到对应连接自己的 mailbox 中。

这样做的目的，是解决头阻塞问题。

效果上：

1. 单个慢客户端主要只影响自己。
2. 主模拟循环不再被串行 socket 发送拖住。
3. Dashboard 观战连接和 bot/玩家连接相互隔离。

`SNAKE_SEND_TIMEOUT_MS` 按“单次 WebSocket 发送”生效。如果某个连接在这个预算内发不出去，该连接会被断开；对玩家来说，这会转入断线保活流程，而不是拖住其他连接。

### 断线保活窗口

玩家连接异常断开后，蛇不会立即从场上移除。

服务端会保留一段可配置的保活窗口。如果玩家在窗口内使用同一个 key 重连，就会恢复到原来的蛇，而不是重新出生。

这样可以缓解课堂 Wi-Fi 短暂抖动带来的体验问题。

### 配置化运行参数

服务端会先从 `config/server.json` 读取运行默认配置。

如果需要临时覆盖，现有环境变量仍然保留且优先级更高；也可以用 `SNAKE_SERVER_CONFIG` 指向另一份 JSON 配置文件。

以下参数都被设计成可配置项：

1. `SNAKE_TICK_RATE`
2. `SNAKE_SEND_TIMEOUT_MS`
3. `SNAKE_DISCONNECT_GRACE_MS`
4. `SNAKE_SPECTATOR_RECONNECT_MS`
5. `SNAKE_MAX_REGISTERED_PLAYERS`
6. `SNAKE_MAX_SPECTATORS`

这样部署者可以根据课堂网络环境调参，而不是修改代码。

新增的两个容量参数主要用于课堂环境的资源保护：

1. 默认限制总注册玩家数，避免错误脚本在一节课内无限堆积注册表。
2. 默认限制观战连接数，避免匿名 spectator 无限占用全量状态广播资源。
3. 任一上限设为 `0` 时，表示关闭该限制。

### 弱网验证实验室

仓库现在在 `docker-compose.full.yml` 里内置了一套 Toxiproxy 弱网实验环境。

这样做有两个目的：

1. 课堂演示时，可以用真实 TCP/WebSocket 链路稳定复现弱网现象。
2. 维护时，可以把 sender 隔离、断线保活和状态可观测性放到端到端环境里回归验证。

这套实验环境会让正常 Dashboard 和直连 bot 继续走服务端直连链路，而被降级的 bot / spectator 走 `toxiproxy:15001`。

这种链路分离很关键，因为它能直接验证我们的目标属性：一条受损链路不应拖住健康链路。

## Dashboard 设计

Dashboard 主要服务两类人：

1. 讲师和学生，需要快速理解当前战局。
2. 运维/组织者，需要判断系统是否正常运行。

当前 Dashboard 展示：

1. 实时游戏画面。
2. 即时长度排行榜。
3. 平均存活长度和平均存活时间统计。
4. 连接状态和事件日志。

之所以增加生存统计，是因为单次历史最长值受运气影响较大，不适合作为 bot 质量的主要比较指标。

## 文档设计

从 `0.2.0` 开始，文档体系固定为四个组成部分：

1. `README`
   作为总入口，面向开发者、课堂组织者和贡献者。
2. `DESIGN`
   解释架构、协议和运维侧的设计取舍。
3. `API`
   描述客户端与服务端之间的接口契约。
4. `CHANGELOG`
   记录每个版本的功能、行为和协议变化。

这个结构的目标，是让未来的维护、追踪和版本沟通更清晰。

## 测试策略

当前自动化验证分成三层：

1. 核心逻辑测试
   验证游戏规则、重生、统计口径等行为。
2. 端到端测试
   验证注册、WebSocket、观战、断线保活、恢复连接等场景。
3. Docker 驱动的弱网端到端测试
   通过真实 Toxiproxy 容器转发 HTTP / WebSocket 流量，验证弱网下的真实行为。

这样拆分，是为了让“规则类问题”、“常规网络问题”和“弱网退化问题”更容易定位。

当前弱网套件重点覆盖：

1. 下行延迟对 HTTP 和 WebSocket 投递的影响。
2. 上行延迟导致控制指令生效变慢，但不会卡住健康观察者。
3. 直连玩家和直连观战端不会被代理慢链路拖住。
4. 代理 reset 后，断线保活窗口内能恢复原蛇。
5. `/status` 在代理故障期间和保活窗口结束后都能正确反映状态。
6. `timeout` 和 `limit_data` 这类黑洞/截断场景。

这套实验室当前也有边界：

1. 它主要覆盖 Toxiproxy 直接支持的传输层退化行为。
2. 还没有覆盖内核级的乱序、损坏或概率型丢包。
3. 如果后续要验证这类场景，更合适的做法是在 Toxiproxy 之外再补一层 `tc netem`，而不是替换掉现有实验室。

## 已知取舍

1. 全量状态快照比增量协议更简单，但带宽成本更高。
2. Dashboard 现在按收到的快照直接渲染，因此视觉流畅度仍受广播频率影响；如果将来需要更平滑，需要再加插值层。
3. 当前实现是按课堂规模优化，而不是按超大规模公网部署优化。
4. 示例客户端现在会对临时重名做有限次数重试，但它仍然只是教学参考实现，不是通用 SDK。
